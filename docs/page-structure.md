## 키패드 조사 결과

- 확인한 이미지 크기: 205×336 CSS pixel, `deviceScaleFactor=1`.
- 숫자/삭제/전체지움/확인 영역은 `<area shape="rect">`로 생성된다.
- 숫자 영역의 `alt`, `title`, `aria-label`, `data-key`는 모두 비어 있었다.
- 1·2·3·4·6은 고정 위치이며 0·5·7·8·9만 다섯 노란 슬롯에서 셔플된다.
- `<area>.boundingBox()`는 0×0이므로 `coords`를 파싱한 뒤 y/x 순서로 정렬한다. 실제 클릭은 전체 페이지가 아니라 키패드 이미지 내부 사각형 중심의 상대좌표를 사용한다.
- 셔플 슬롯은 각 버튼 중앙을 crop하고 grayscale, threshold 175, 흰 여백 trim, 24×36 contain resize를 적용한다.
- 다섯 슬롯과 다섯 템플릿의 픽셀 거리를 계산하고 120개 순열 중 총점이 최소인 조합만 사용한다.
- 평균 픽셀 거리 허용값은 0.12 이하, 최적/차선 총점 차이는 0.025 이상이다. 숫자 중복, 슬롯 수 오류, 버튼 50~60px 범위 이탈 시 `KEYPAD_RECOGNITION_ERROR`로 중단한다.
- 템플릿과 fixture에는 공개 키패드 숫자만 있으며 계좌정보, 비밀번호, 거래정보가 없다.
- 은행 JavaScript의 암호화 값, onclick 데이터, 숨겨진 키 값은 분석하거나 사용하지 않는다.

## 결과 화면 확인 전 필수 작업

실제 운영 전에는 사용자가 로컬 headed 환경에서 단 한 번 정상 조회하고, 민감정보를 제거한 뒤 다음을 확인해야 한다.

1. `#b028770` 내부의 거래 공통 부모 구조와 `#b048488` 대체 후보 여부
2. 실제 헤더 명칭 및 열 순서
3. 거래 없음·인증 오류·점검 문구
4. 페이지네이션 또는 더보기 존재와 전체 건수
5. 결과가 별도 iframe에 표시되는지 여부

이 검증이 끝나기 전에는 결과 파서가 성공했다고 간주하지 않는다.

## 2026-07-15 실제 제출 검증 결과

- Linux Docker headless 환경에서 키패드 인식과 입력 자릿수 검증을 통과한 뒤 조회 버튼을 1회 제출했다.
- 제출 후 35초 동안 현재의 결과 선택자 `#b048488` 및 오류 선택자 `#errorDiv`, `#errMsg`로 완료 상태를 식별하지 못해 `timeout`으로 종료했다.
- 결과 컨테이너에 진입하지 못했으므로 실제 거래 DOM, 거래 건수, 페이지네이션, 더보기, iframe 여부는 확인되지 않았다.
- 개인정보 제거 구조 캡처는 결과 컨테이너 확인 후에만 실행되므로 fixture 파일은 생성되지 않았다.
- 인증 실패인지, 결과 선택자 변경인지, 별도 창/프레임 전환인지 판정할 근거가 없다. 안전 원칙에 따라 같은 인증정보로 재제출하지 않았다.
- 따라서 `#b048488`과 결과 관련 선택자는 여전히 미검증 상태이며, 거래 파서 완성으로 간주하지 않는다.

## 2026-07-15 6개월 거래 존재 기간 검증

- 조회 기간은 `2026-01-15`부터 `2026-07-15`까지이며 headless 제출은 1회만 수행했다.
- `#b028770 table.tType01 tbody`에서 visible 행 4개가 관찰됐다.
- 행 구조는 `8개 td의 본거래 행`과 `1개 td 및 보조 헤더를 가진 상세 행`이 번갈아 나타나는 형태다.
- 따라서 화면의 실제 거래 반복 건수는 2건이며 상세 행 2개는 별도 거래가 아니다.
- 기존 진단은 모든 visible `tr`을 거래로 세어 4건으로 보고했고, 파서는 1셀 상세 행에서 구조 오류로 중단했다.
- 결과 진단과 파서는 이제 헤더 8열과 일치하는 본거래 행만 거래로 세고, 검증된 상세 행은 보조 행으로 제외한다.
- 이번 결과에는 다음/이전/더보기나 pagination 구조가 없었다.
- 수정 후 파서는 개인정보 제거 fixture의 `본거래 → 상세 → 본거래 → 상세` 구조에서 2건 파싱 및 입출금 정규화 테스트를 통과했다.
- 안전 원칙에 따라 수정 후 실제 은행 조회를 추가 제출하지 않았으므로, 실제 값으로 생성된 비어 있지 않은 `Transaction[]`은 아직 종단 재검증되지 않았다.

- 상세 행의 논리 열 수, 본거래 직후 위치와 거래별 1:1 반복 관계를 검증한다. 알려진 송금인·수취인·중립 역할은 거래 방향에 따라 혼합될 수 있지만, 역할이 불명확하면 계속 중단한다.
